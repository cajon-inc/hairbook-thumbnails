# サロンページ掲載画像を活用したMetaバナー制作・配信計画

- 更新日: 2026-07-24
- 対象: Hairbook 集客カタログ（HB_02）
- 目的: 各サロンページに掲載されたモデル／スタイル画像と店内写真を素材として選定し、画像補正・4:5最適化・正確なテキスト合成を行い、サロン別のMetaバナーとして比較・承認・配信・効果測定する。
- 本書は、従来の「架空人物をゼロベース生成する」方針に代わる正本とする。
- **前提**: 本計画の対象となるサロンページ内の人物画像・店内写真は、Meta広告利用、背景変更・補正・トリミング・テキスト追加を含む編集利用が可能な素材として提供される。個別の権利確認工程は本計画のスコープに含めない。

> **2026-07-24 全体展開方針**
> 全静止画CRの標準は、人物画像版V3レイヤードとする。店内写真版は比較用試作として保持するが、通常の全件展開候補にはせず、人物sourceを用意できない場合の例外承認に限定する。差し替え手順と必須事前チェックは `STATIC_CATALOG_PERSON_V3_ROLLOUT_PLAN.md` を正本とする。

## 1. 結論

採用する基本方針は次のとおり。

1. 主画像は、対象サロンの公式ページ、Hairbookの対象サロンページ、または対象店舗の掲載ページにあるモデル／スタイル画像から選ぶ。
2. 対象画像はすべて利用可能という前提で、画質・構図・訴求との一致を基準にsourceを選定する。
3. 人物・内装をゼロベース生成せず、選定した既存画像を入力にした**画像編集**として扱う。人物の同一性や実際の内装を不用意に変更しない。
4. 編集範囲は、背景変更・背景整理、露出／色調／ノイズ等の補正、4:5へのクロップまたはアウトペイント、テキストを置く余白の確保を基本とする。
5. 店舗名、エリア、アクセス、訴求文、CTAは画像生成モデルへ描かせず、確認済みデータからコードで合成する。
6. 全体展開では1サロンにつき人物画像版V3を標準とする。人物sourceがない場合は現行画像を維持し、店内写真版は例外承認を受けたときだけ使用する。
7. 自動編集から自動配信へ直結させない。画像QA、事実確認、人手承認を通過した最終バナーだけを公開する。
8. 画像切替履歴を不変ログとして残し、Metaの日次 `product_id` 実績と結び付け、現在画像／人物画像版V3の配信結果を振り返る。
9. 店内写真版を例外検証する場合も、主画像種別以外を固定する。店舗情報、訴求、CTA、対象、計測ロジックを同時に変えない。

## 2. 方針変更

### 変更前

- サロンの確認済みテキストを根拠に、架空の成人モデルをゼロベース生成する。
- 公式ページの人物画像はモデル入力に使わない。

### 変更後

- 対象サロンページに掲載されたモデル／スタイル画像と店内写真を各1枚まで選定し、それぞれ補正・4:5最適化を行う。
- 対象画像は利用可能な素材として提供される前提とし、権利確認待ちを挟まず編集パイロットへ進む。
- 最終バナーの文字はコード合成とし、正確な店名・アクセス・訴求を保証する。

### 変更の狙い

- サロンが実際に掲載している人物・スタイルを使い、サロンとの関連性を強める。
- 過去の比較試作として店内写真版を保持し、例外利用を判断するときの参考にする。
- 架空人物によるブランド不一致や「このサロンの実例か」という誤認を減らす。
- 元写真の髪型・質感を活かしながら、広告面で読みやすい背景と情報設計へ整える。

## 3. 使用できる掲載画像

### 優先順位

| 優先 | 画像種別 | 初期方針 |
|---:|---|---|
| 1 | サロンページのスタイル／モデル写真 | 髪型・質感・サロンの得意領域が分かるものを優先 |
| 2 | 対象店舗の店内写真 | セット面、受付、照明、清潔感など店舗の空間が判別できる全景を優先 |
| 3 | サロンが管理する施術イメージ写真 | 訴求内容と一致し、4:5へ展開しやすいものを採用 |
| 4 | スタッフ写真 | スタッフ訴求と一致し、広告の主役として成立する場合に採用 |
| 5 | その他のページ内画像 | 利用文脈と訴求が一致する場合に採用 |

### 本計画での利用前提

対象サロンページ内の人物画像・店内写真は、次の利用が可能な状態で提供されるものとする。

- Meta有料広告での使用。
- 背景変更、背景整理、トリミング、色調補正、テキスト追加。
- AIを用いた人物以外の背景編集またはアウトペイント。
- Hairbookのクリエイティブ管理画面と配信履歴での保存・表示。

システム上は `usage_status="available"` として一括登録し、サロンごとの個別確認待ち状態は設けない。素材の利用条件が別途変更された場合だけ対象assetを停止し、直前画像へ戻す。

### クリエイティブ上、選定しない画像

- 対象サロン以外の人物・スタイル・店舗写真が混在している画像。
- 実際の施術結果と誤認させる前後比較、医療的効果、誇大表現を含む画像。
- 元画像の解像度が低く、4:5編集後に髪・顔を明瞭に保てない画像。
- 透かし、第三者ロゴ、読める不要文字が大きく入り、広告面として整理しにくい画像。
- 求人・採用・イベントの文脈が強く、集客訴求と一致しない画像。
- 内装の一部だけで対象店舗だと判断できない写真、人物が大きく写り店内が判別できない写真。

## 4. 画像取得と正本

### 情報源

| 情報 | 正本 |
|---|---|
| salon_id / salon_name | フィード生成元DB |
| canonical salon URL / landing URL | フィード生成元DBと実着地ページ |
| モデル／スタイル・店内写真候補 | 対象サロンページまたは対象店舗掲載ページ内の画像 |
| エリア・アクセス | DBおよび対象サロンページで一致確認 |
| 訴求 | 対象サロンページで確認できる得意領域・サービス |
| 利用状態 | 本計画の前提に基づき `available` |

### 取得時に保存する情報

```json
{
  "salon_id": "<private salon id>",
  "salon_name": "<canonical name>",
  "source_page_url": "https://...",
  "source_image_url": "https://...",
  "source_image_hash": "<sha256>",
  "source_page_hash": "<sha256>",
  "fetched_at": "<ISO-8601>",
  "source_type": "model_style|salon_interior|staff|other",
  "person_type": "model|customer|staff|unknown",
  "usage_context": "style|salon|staff|recruit|other",
  "usage_status": "available",
  "usage_assumption_version": "2026-07-24",
  "readiness": {
    "status": "ready|quality_hold|source_mismatch",
    "reasons": []
  }
}
```

- 元画像とsource metadataはprivate側へ保存する。
- public側には承認済み最終バナーと匿名アセットIDだけを置く。
- publicリポジトリへ人物名、元ページ本文、内部ID対応表を置かない。
- source画像がページから削除・変更されても過去の配信根拠を追跡できるよう、hashと取得時刻を残す。

## 5. 編集仕様

### 編集の基本

画像編集は、元人物または実際の店内を維持したまま広告面へ最適化する。

1. 元画像を4:5の1080×1350へ配置する。
2. モデル画像版では顔・髪・主要スタイル、店内写真版ではセット面・受付・空間の特徴をセーフエリアに保持する。
3. 背景を整理し、文字エリアのコントラストを確保する。
4. 露出、ホワイトバランス、色かぶり、ノイズ、シャープネスを必要最小限補正する。
5. 必要に応じて、同一写真のぼかし背景または人物以外の背景だけのアウトペイントで4:5へ適合する。
6. 編集済み背景画像を確定した後、コードで店舗情報と訴求を合成する。

### 許可する編集

- 背景のぼかし、整理、不要な背景物の除去。
- サロンのトーンに合わせた非特定背景への変更。
- 背景のみのアウトペイント。
- 露出、色温度、彩度、コントラスト、ノイズ、解像感の補正。
- 4:5クロップと文字用余白の確保。
- 衣服の軽微な明るさ・色調補正。
- 正確な店舗情報、訴求、CTAのコード合成。

### 原則禁止する編集

- 顔の差し替え、別人化、年齢・人種・性別表現の変更。
- 髪型、髪色、髪の長さ、髪質を別スタイルへ変更すること。
- 体型、肌、顔立ちを大きく変える美容加工。
- スタッフ写真を顧客の施術例のように見せること。
- 実店舗ではない背景を実店舗だと表現すること。
- 店内写真の壁、什器、間取り、店舗ロゴを別店舗のように改変すること。
- 透かし、著作権表示、第三者ロゴの除去。
- 元ページにない施術効果、価格、受賞、順位、期間限定情報の追加。

人物、髪、顔に修復が必要な場合は、原則として別の元画像を選ぶ。生成的な修復を行う場合は範囲を記録し、人手で同一性と誤認を確認する。

### テキスト仕様

表示順は全サロンで次を基本とする。

1. エリア
2. 店舗名
3. アクセス
4. 主訴求
5. 補足訴求
6. CTA

文字は生成画像へ焼き込ませず、フォント・文字列・改行・位置をコードで決定する。これにより、日本語、駅名、徒歩分数、店名の誤記を防ぐ。

### Meta 4:5の視認性基準

1080×1350pxの最終バナーでは、次を初期基準とする。

| 情報 | 基準サイズ |
|---|---:|
| エリア | 34px以上 |
| 店舗名 | 52px以上 |
| アクセス | 36px以上 |
| 主訴求 | 56px以上 |
| 補足訴求 | 34px以上 |
| CTA | 38px以上、ボタン高94px以上 |

- 左右の基本余白を56px以上とし、各情報の開始位置を揃える。
- 写真下部へ高コントラストの情報面を設け、写真上へ細い文字を直接置かない。
- 長いアクセス情報は文字を小さくせず、意味の切れ目で2行にする。
- 360×450px相当へ縮小し、エリア、店舗名、アクセス、主訴求、CTAが判読できることを人手QAへ追加する。

### レイヤードデザイン基準

単色の情報面で画面を上下に二分すると平面的に見えるため、承認用の標準案では次を優先する。

- 写真を画面全体へ連続させ、文字領域だけ段階スクラムで沈める。
- 店舗名プレート、アクセス枠、主訴求、CTAを別レイヤーとして扱い、影・透過率・罫線で前後関係を作る。
- 明朝とゴシックの2書体、太線と細線、明部と暗部を組み合わせ、すべてを同じ強さにしない。
- 店内写真版もモデル画像版と同じ情報設計を使い、写真種別以外の差を増やさない。
- 過去のFier 大阪梅田案をレイヤード表現の基準とし、V3では外周フレーム、半透明プレート、段階スクラム、アクセント罫線を共通化する。

## 6. 候補と状態管理

### 管理画面の切替候補

| 表示 | 内容 | 切替可否 |
|---|---|---|
| 現在画像 | 比較とロールバック用 | 復元時のみ |
| 人物画像版V3 | 選定済みモデル／スタイル画像を使う標準案 | 可 |
| 店内写真版 | 人物sourceがない場合の例外案 | 例外承認時のみ |

- 人物または例外の店内sourceを変えた案を同じslotへ上書きしない。source versionを増やし、再承認する。
- 過去の比較試作では両版の文字要素を共通にしたが、全件展開時の標準slotは人物画像版V3の1案とする。
- 再編集では既存候補を上書きせずversionを増やす。

### 状態遷移

```text
source_discovered
  → source_selected
  → editing
  → edited
  → qa_failed | qa_passed
  → approved | rejected
  → published
  → retired
```

- `source_selected` であれば編集ジョブを開始できる。
- `qa_passed` だけでは配信不可。`approved` に承認者と承認日時を必須にする。
- source画像が変わったら既存承認を `stale` として表示し、自動更新しない。

### 編集アセット記録

```json
{
  "asset_id": "<uuid>",
  "salon_id": "<private mapping>",
  "source_asset_id": "<private source image id>",
  "source_image_hash": "<sha256>",
  "candidate_slot": "model_image|salon_interior",
  "asset_version": 1,
  "representation": "salon_page_person_image_edit|salon_interior_photo",
  "edit_spec_version": 1,
  "edit_prompt_hash": "<sha256>",
  "edit_operations": ["background_replace", "crop_4x5", "color_correct"],
  "copy_version": 1,
  "overlay_version": 1,
  "source_record_hash": "<sha256>",
  "edited_at": "<ISO-8601>",
  "qa": {},
  "approval": {},
  "public_image_url": ""
}
```

## 7. QAと承認

### 自動QA

| 検査 | 合格条件 |
|---|---|
| ファイル | JPEG、1080×1350、sRGB、Exif除去、容量上限内 |
| source一致 | source hash、対象サロン、source pageが一致 |
| source同一性 | 人物の顔・髪型・髪色、または店内の什器・間取り・ロゴが元画像から不自然に変わっていない |
| 構図 | 顔・髪・主要スタイル、または店内の主要空間がセーフエリア内 |
| 背景 | 破綻、重複、看板、第三者ロゴ、読める不要文字がない |
| OCR | コード合成前の編集画像に不要な文字・数字がない |
| テキスト | 店名、エリア、アクセス、訴求が正本と一致 |
| 誤認 | 実店舗でない背景を実店舗として主張していない |
| 広告適性 | 集客広告であり、求人・告知の混入がない |

### 人手確認

1. 元画像と編集後画像を横並びで確認する。
2. 対象サロンとモデル／店内画像の対応が正しい。
3. 顔、髪型、髪色、体型、または店内の実在要素が不自然に変わっていない。
4. 背景変更が広告表現として誤認を生まない。
5. 店名、エリア、アクセス、訴求が着地先と一致する。
6. 4:5と見切れプレビューで主要要素が残る。
7. 現在画像へ戻せることを確認する。

## 8. 管理画面

「画像設定」タブには次を表示する。

- 元サロンページURLと取得日時。
- 元モデル画像・元店内写真と編集後画像の比較。
- source種別、人物種別、利用文脈、source選定状態。
- 背景変更・補正・クロップ等の編集内容。
- 人物画像版V3。人物sourceがない場合だけ、例外承認された店内写真版を追加する。
- 最終テキスト合成プレビュー。
- 自動QA、人手承認、却下、再編集、公開、ロールバック。

「配信結果」タブには次を表示する。

- 現在画像／人物画像版V3／例外の店内写真版の公開期間。
- source version、asset version、copy／overlay version。
- spend、impressions、clicks、CTR、CPM、Meta実測 `add_to_cart`。
- 切替前後、treatment／holdoutの比較。店内版を例外配信した場合だけ画像種別比較を追加する。
- 完全日、除外日、最小impressions到達状況。

一覧カードでは `source_selected / quality_hold / qa_failed / approved / published / rollback_required` を絞り込めるようにする。

## 9. パイプライン

```text
[サロンページ／店舗掲載ページ]
      ↓ モデル／スタイル・店内画像候補・source hash
[source選定 / usage available]
      ↓ source_selected
[背景編集・補正・4:5最適化]
      ↓ edited asset
[コードで店舗情報・訴求を合成]
      ↓ final banner
[QA・人手承認]
      ↓ approved public JPG
[thumbnail_override → Meta]
      ↓
[publish events × product_id日次実績]
      ↓
[/creative 配信結果]
```

### API案

- `POST /api/creative/:salon_id/discover-source` … サロンページのモデル／スタイル画像・店内写真候補を取得
- `POST /api/creative/:salon_id/select-source` … 利用可能な候補から編集元を確定
- `POST /api/creative/:salon_id/edit` … source_selected画像を人物画像版V3として処理。例外承認時だけ店内写真版を許可
- `POST /api/creative/:salon_id/approve` … 元画像・編集差分・最終バナーを承認
- `POST /api/creative/:salon_id/publish` … final banner保存、publish event作成、enrich起動
- `POST /api/creative/:salon_id/rollback` … 直前overrideを復元
- `GET /api/creative/:salon_id/performance` … asset version別実績

### `creative_config` 例

```json
{
  "<product_id>": {
    "enabled": true,
    "source": "edited_banner",
    "source_url": "https://raw.githubusercontent.com/.../<anonymous-hash>.jpg",
    "source_params": {
      "asset_id": "<anonymous asset id>",
      "candidate_slot": "pattern_1",
      "asset_version": 1,
      "representation": "salon_page_person_image_edit"
    }
  }
}
```

最終バナーに文字を合成済みの場合は、`enrich.py` が帯を二重合成しない `render_mode="complete_banner"` を追加する。背景編集済み画像へ本番側で帯を付ける場合は `render_mode="overlay"` とする。初期パイロットでどちらか1方式へ固定する。

## 10. 効果測定

引き継ぎの定義を変えない。

- 対象広告アカウント: HB_01／HB_02／HB_07。カタログ実験はHB_02。
- 主指標: CTR／CPM。
- 従指標: Meta実測 `add_to_cart`。
- カタログBotのメタCPA／CV: 絶対値ではなくトレンドのみ。
- 割付単位: サロン。1サロンをtreatmentとholdoutへ混在させない。
- 例外として人物画像版と店内写真版を比較する場合は、主画像種別以外の店舗情報、訴求、CTA、配信対象、計測ロジックを固定する。
- 切替日を除外し、最初の完全なMetaアカウント日から集計する。

運用指標には次を追加する。

- モデル／スタイル画像・店内写真候補取得率。
- source選定完了率。
- 編集QA合格率。
- 人手承認率。
- source差し替え率。
- 1承認バナーあたり編集コスト。
- 1サロンあたりsource選定・編集・承認時間。

### 人物画像版と例外の店内写真版の比較

- 比較単位は `salon_id × asset_version × 完全日` とする。
- `creative_variant_type` に `model_image` または `salon_interior` を保存する。
- 人物画像版は施術後イメージ、例外の店内写真版は雰囲気・清潔感・来店時の安心感を主な仮説とする。
- 主指標CTR／CPM、従指標Meta実測 `add_to_cart` を共通定義で集計する。
- 同一サロン内で同日に両版が混在した日は比較から除外する。
- 最小impressionsと観測日数は配信前に決め、母数未達は勝敗判定せず参考値として表示する。

## 11. 段階導入

### Phase 0 — 利用前提・編集ルール確定

- 対象サロンページ内の人物画像は利用可能という前提を関係者で共有する。
- スタイル写真、モデル写真、顧客写真、スタッフ写真の選定基準を定義する。
- 背景変更、肌・髪補正、アウトペイントの許容範囲を承認する。
- 完了条件: usage assumption version、選定基準、禁止編集、承認者が文書化されている。

### Phase 1 — 8サロンのsource選定

- 現在の試作3店舗を含む8サロン程度でモデル／スタイル画像と店内写真の候補を取得する。
- source page、source image hash、source種別、人物種別、利用文脈、画質を記録する。
- 画質・構図・訴求との一致を基準に各種別1枚まで選定する。
- 完了条件: 対象サロンごとに人物画像版V3の `source_selected` 画像が1枚確定。確定できないサロンは `person_source_hold`。

### Phase 2 — 編集パイロット

- source_selected画像から人物画像版V3を1案作る。店内版は例外承認されたサロンだけ作る。
- 背景変更、補正、4:5最適化、コード合成文字を確認する。
- 実配信せず、元画像との同一性、誤認、品質、作業時間を評価する。
- 完了条件: 重大な人物変形、背景破綻、店舗情報誤記が0件。

### Phase 3 — 公開・計測基盤

- 認証、private source metadata、source／asset version、QA、承認、publish event、ロールバックを実装する。
- `complete_banner` と `overlay` の二重合成防止を実装する。
- Meta `product_id` 日次実績を画像有効期間へ時系列結合する。

### Phase 4 — HB_02小規模配信

- 過去実績から対象数、最小impressions、観測期間を事前に決める。
- サロン単位treatment／holdoutで固定割付する。
- 切替日を除外し、完全日だけを比較する。
- 問題時はサロン単位で直前overrideへ戻す。

### Phase 5 — 段階拡張

- 広告効果と運用負荷が許容範囲のサロンから拡張する。
- 元画像削除、人物画像差し替え、スタッフ退職等を再レビューキューへ入れる。

## 12. 最初に実行する作業

1. 8サロン分のcanonical URLと実着地URLを取得する。
2. 各ページ内の画像候補を一覧化し、スタイル／モデル／顧客／スタッフ／店内／求人を分類する。
3. 画質・構図・訴求との一致から、モデル画像と店内写真を各1枚まで選ぶ。
4. 選定画像をダウンロードし、source hashと取得日時をprivate保存する。
5. 補正・4:5最適化した人物画像版V3を作る。人物sourceがない場合は保留し、店内版を自動生成しない。
6. 店舗名、エリア、アクセス、訴求、CTAをコードで合成する。
7. 元画像・編集画像・最終バナーを同じ画面でQAし、人手承認する。
8. 8件のsource選定完了率、編集QA合格率、再編集率、作業時間をレビューする。
9. 本番基盤とロールバックが整ってからHB_02小規模実験へ進む。
